CIA Triad - Foundation of Information Security

The Three Pillars of Information Security

The CIA Triad forms the foundation of information security, providing a framework for evaluating and implementing security controls. Every security decision should consider how it impacts Confidentiality, Integrity, and Availability.

Confidentiality

Information is accessible only to authorized users. Prevents unauthorized disclosure of sensitive data through access controls, encryption, and classification.

SOC Implementation:

  • • Data classification policies
  • • Encryption at rest and in transit
  • • Access controls and permissions
  • • Data loss prevention (DLP)
  • • Network segmentation

Integrity

Information remains accurate and unaltered. Ensures data hasn't been modified by unauthorized parties through checksums, digital signatures, and version control.

SOC Implementation:

  • • File integrity monitoring (FIM)
  • • Digital signatures and hashing
  • • Version control systems
  • • Change management processes
  • • Database transaction logs

Availability

Information and systems are accessible when needed by authorized users. Prevents service disruption through redundancy, monitoring, and disaster recovery.

SOC Implementation:

  • • Redundancy and failover systems
  • • DDoS protection and monitoring
  • • Backup and disaster recovery
  • • Performance monitoring
  • • Incident response procedures

Common Threats Against CIA Triad

CIA Element Common Threats SOC Detection Methods
Confidentiality Data breaches, eavesdropping, insider threats DLP alerts, access anomalies, encryption monitoring
Integrity Malware, unauthorized modifications, tampering FIM alerts, hash verification, change detection
Availability DDoS, ransomware, system failures Performance monitoring, service health checks

Authentication Mechanisms

Authentication vs Authorization

Authentication

"Who are you?"

  • • Verifies identity
  • • Proves you are who you claim to be
  • • Uses credentials (password, token, biometric)
  • • Happens first in access control

Authorization

"What can you access?"

  • • Determines permissions
  • • Controls what you can do
  • • Based on roles, policies, attributes
  • • Happens after authentication

Authentication Factors

Something You Know

  • • Passwords and passphrases
  • • PINs and security questions
  • • Knowledge-based authentication

Something You Have

  • • Hardware tokens and smart cards
  • • Mobile devices and SMS
  • • Software tokens and authenticator apps

Something You Are

  • • Fingerprints and facial recognition
  • • Iris and retina scans
  • • Voice and behavioral biometrics

Multi-Factor Authentication (MFA)

# Common MFA implementations # TOTP (Time-based One-Time Password) Google Authenticator, Authy, Microsoft Authenticator # Hardware tokens RSA SecurID, YubiKey # Push notifications Duo Security, Microsoft Authenticator push # Biometric + Password Windows Hello, TouchID + password

MFA Bypass Techniques (SOC Awareness):

  • SIM swapping attacks
  • MFA fatigue attacks
  • Session hijacking post-authentication
  • Social engineering of helpdesk
  • Malware intercepting SMS codes

Authorization Models

Role-Based Access Control (RBAC)

Access permissions are assigned to roles, and users are assigned to roles based on their responsibilities.

# RBAC Example Role: Database_Admin Permissions: - CREATE_DATABASE - DELETE_DATABASE - BACKUP_DATABASE - MANAGE_USERS User: john_doe Assigned_Roles: [Database_Admin, Security_Auditor]

RBAC Benefits:

  • • Simplifies permission management
  • • Scalable for large organizations
  • • Principle of least privilege
  • • Easy role-based reporting

Attribute-Based Access Control (ABAC)

Access decisions based on attributes of users, resources, environment, and actions.

# ABAC Policy Example PERMIT IF ( user.department == "Finance" AND resource.classification == "Confidential" AND environment.location == "Corporate_Network" AND environment.time BETWEEN 09:00 AND 17:00 )

ABAC Advantages:

  • • Fine-grained access control
  • • Dynamic policy evaluation
  • • Context-aware decisions
  • • Supports complex requirements

Advanced Access Control Models

Mandatory Access Control (MAC)

System-enforced access control based on security labels and clearance levels. Users cannot override system policies.

# MAC Implementation Examples # SELinux (Security-Enhanced Linux) ls -Z /etc/passwd # Shows SELinux context: user:role:type:level # Security Levels (Bell-LaPadula Model) Top Secret (Level 4) Secret (Level 3) Confidential (Level 2) Unclassified (Level 1) # "No Read Up, No Write Down" Rule # Users can only: # - Read at their level or below # - Write at their level or above

MAC Characteristics:

  • • System administrator controls all access
  • • Users cannot change object classifications
  • • Common in government/military environments
  • • Prevents information leakage to lower levels
  • • Examples: SELinux, AppArmor, Windows MIC

Discretionary Access Control (DAC)

Resource owners have discretion over who can access their resources. Users can grant or revoke access to objects they own.

# DAC Implementation Examples # Linux File Permissions chmod 755 myfile.txt # Owner: rwx, Group: rx, Other: rx chown user:group myfile.txt # Change ownership # Windows NTFS Permissions icacls file.txt /grant user:F # Grant full control icacls file.txt /remove user # Remove user access # Access Control Lists (ACLs) # Linux Extended ACLs setfacl -m u:john:rw file.txt # Grant read/write to john getfacl file.txt # View ACL # Windows ACLs # Right-click → Properties → Security → Edit

DAC Examples & Use Cases:

  • • File system permissions (NTFS, ext4)
  • • Database object permissions
  • • Application-level access controls
  • • Cloud storage sharing (Google Drive, OneDrive)
  • • Most common in business environments

Cryptography & Encryption

Cryptography Fundamentals

Cryptography provides the mathematical foundation for protecting data confidentiality, integrity, and authenticity. Understanding encryption methods is essential for SOC analysts to identify proper implementations and detect cryptographic attacks.

Symmetric Encryption

Same key used for both encryption and decryption. Fast and efficient for large amounts of data, but requires secure key distribution.

# Common Symmetric Algorithms AES (Advanced Encryption Standard) - AES-128, AES-192, AES-256 - Block cipher, 128-bit blocks - Most widely used 3DES (Triple DES) - Legacy, being phased out - 112-bit effective key strength ChaCha20 - Stream cipher - Used in TLS 1.3, SSH # AES Example (OpenSSL) echo "sensitive data" | openssl enc -aes-256-cbc -base64 -pass pass:mykey # File encryption openssl enc -aes-256-cbc -salt -in file.txt -out file.enc -pass pass:mypassword

Asymmetric Encryption

Uses a pair of mathematically related keys (public/private). Solves key distribution problem but slower than symmetric encryption.

# Common Asymmetric Algorithms RSA (Rivest-Shamir-Adleman) - Key sizes: 2048, 3072, 4096 bits - Most common for key exchange ECDSA (Elliptic Curve Digital Signature) - Smaller keys, same security - EC P-256, P-384, P-521 Ed25519 - Modern, fast elliptic curve - SSH and TLS implementation # RSA Key Generation openssl genrsa -out private.pem 2048 openssl rsa -in private.pem -pubout -out public.pem
